# bug_fix 评分参考

适用场景：用户提供 Bug 修复 diff、修复前后代码、问题描述与修复结果，目标是评价修复是否命中问题且控制侵入范围。

## 一级维度与权重

| 一级维度 | 权重 | 核心关注点 |
|---|---:|---|
| 改动精准度与最小侵入性 | 25 | 是否命中问题、修改范围克制、无关重构控制 |
| 代码正确性与静态质量 | 20 | ArkTS/ArkUI 语法、类型安全、明显错误 |
| 与既有工程一致性 | 15 | 命名风格、结构接入、原模式复用 |
| 风险控制与稳定性 | 15 | 回归风险、稳定性风险、安全/边界意识 |
| 平台规范符合度 | 10 | ArkTS/ArkUI 规范、HarmonyOS 工程实践 |
| 可维护性与可读性 | 10 | 修改点可理解性、复杂度、可演进性 |
| 架构与职责影响控制 | 5 | 是否破坏原边界、是否引入额外耦合 |

## 二级指标

### 改动精准度与最小侵入性（25）
- 问题点命中程度（10）
- 修改范围克制度（8）
- 无关重构控制（7）

### 代码正确性与静态质量（20）
- ArkTS/ArkUI语法与类型安全（8）
- 静态坏味道控制（6）
- 明显错误密度（6）

### 与既有工程一致性（15）
- 命名与风格一致性（5）
- 结构接入一致性（5）
- 原模式复用程度（5）

### 风险控制与稳定性（15）
- 回归风险控制（6）
- 稳定性风险（5）
- 安全/边界意识（4）

### 平台规范符合度（10）
- ArkTS/ArkUI规范符合度（5）
- HarmonyOS工程实践符合度（5）

### 可维护性与可读性（10）
- 修改点可理解性（4）
- 复杂度控制（3）
- 可演进性（3）

### 架构与职责影响控制（5）
- 是否破坏原边界（3）
- 是否引入额外耦合（2）

## 细化评分表

说明：二级指标按下表离散档位给分，bug_fix 场景优先看“是否命中问题、是否过修、是否引入新风险”；若根因证据不足，应降低置信度并进入人工复核。

### 改动精准度与最小侵入性

| 二级指标 | 分值 | 给分标准 |
|---|---:|---|
| 问题点命中程度 | 10 | 修改直接命中根因或完整故障链路，修复点与问题描述、现象、上下文高度一致，没有靠碰运气规避问题。 |
| 问题点命中程度 | 8 | 明确命中主要问题点，根因判断基本成立，仅少量链路证据不够充分。 |
| 问题点命中程度 | 6 | 修复能覆盖主要症状，且大概率触达真正原因，但根因闭环不够完整。 |
| 问题点命中程度 | 3 | 主要是在症状层做补丁，是否真正解决问题仍不稳定。 |
| 问题点命中程度 | 0 | 改动与问题点明显不匹配，属于误修或未修到位。 |
| 修改范围克制度 | 8 | 改动只触及解决问题所必需的代码路径，文件数、修改面和影响半径都控制得很克制。 |
| 修改范围克制度 | 6 | 范围总体克制，存在少量可解释的附带修改。 |
| 修改范围克制度 | 4 | 改动范围偏大，但大部分仍与修复链路相关。 |
| 修改范围克制度 | 2 | 修改明显扩散，已经出现过修迹象。 |
| 修改范围克制度 | 0 | 改动范围严重失控，接近或达到 G4。 |
| 无关重构控制 | 7 | 没有引入与修复无关的重构、风格清理或结构调整，diff 噪音极低。 |
| 无关重构控制 | 5 | 仅有少量无关优化，但不影响问题判断和 review。 |
| 无关重构控制 | 3 | 存在较明显的顺手修改，需要额外甄别主次。 |
| 无关重构控制 | 1 | 无关重构占比偏高，掩盖修复主体。 |
| 无关重构控制 | 0 | diff 主要被无关调整占据，难以确认真实修复内容。 |

### 代码正确性与静态质量

| 二级指标 | 分值 | 给分标准 |
|---|---:|---|
| ArkTS/ArkUI语法与类型安全 | 8 | 修复后的改动无明显 ArkTS/ArkUI 语法问题，类型关系稳定，没有靠弱类型逃逸完成修复。 |
| ArkTS/ArkUI语法与类型安全 | 6 | 存在少量边界性类型问题，但不影响整体修复质量。 |
| ArkTS/ArkUI语法与类型安全 | 4 | 有多处类型或语法可疑点，修复质量需要补强。 |
| ArkTS/ArkUI语法与类型安全 | 2 | 通过引入较多类型问题、未定义引用或不合规写法完成修补。 |
| ArkTS/ArkUI语法与类型安全 | 0 | 高密度静态错误或明显违反 ArkTS 基本约束。 |
| 静态坏味道控制 | 6 | 修复实现保持简洁，没有新增重复逻辑、死代码、魔法值堆积或无意义包装。 |
| 静态坏味道控制 | 4 | 有少量坏味道，但不影响主要修复判断。 |
| 静态坏味道控制 | 2 | 为修 bug 引入了较多坏味道，后续债务明显。 |
| 静态坏味道控制 | 0 | 修复方式本身制造了密集坏味道。 |
| 明显错误密度 | 6 | 修复改动几乎没有新的明显错误，单点问题不形成模式。 |
| 明显错误密度 | 4 | 有少量明显错误，但局部且容易修补。 |
| 明显错误密度 | 2 | 多处新增错误或旧问题未消除，可信度明显下降。 |
| 明显错误密度 | 0 | 错误密度高，修复产物整体不可靠。 |

### 与既有工程一致性

| 二级指标 | 分值 | 给分标准 |
|---|---:|---|

| 命名与风格一致性 | 4 | 主体一致，仅少量局部风格差异。 |
| 命名与风格一致性 | 3 | 基本可接受，但新增片段略显游离。 |
| 命名与风格一致性 | 2 | 风格差异明显，维护者需要额外适应。 |
| 命名与风格一致性 | 1 | 命名和风格偏离较大。 |
| 命名与风格一致性 | 0 | 修复代码与原工程明显割裂。 |
| 结构接入一致性 | 5 | 修复发生在原本最合理的位置，未为了修 bug 新建不必要层级或绕开既有结构。 |

| 结构接入一致性 | 2 | 修复位置不理想，局部破坏原结构秩序。 |

| 原模式复用程度 | 1 | 大量重复或旁路实现。 |
| 原模式复用程度 | 0 | 基本无视原工程模式。 |

|---|---:|---|

| 是否破坏原边界 | 2 | 有轻微边界泄漏，但尚可接受。 |

| 是否引入额外耦合 | 0 | 引入明显额外耦合，后续问题定位会更困难。 |

## 本类型常见风险
- 没有问题描述或参考上下文，却给出高置信结论。
- 把“修复完成”误当成“工程质量高”。
- 因顺手优化造成 diff 噪音和误修。
- 在小问题上引入大范围结构改造。

## 输出时应强调
- 是否命中问题点。
- 是否存在误修、过修、无关重构。
- 是否触发 G4 硬门槛候选。
- 回归风险与边界风险是否被控制。
