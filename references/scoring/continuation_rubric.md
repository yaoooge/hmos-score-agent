# continuation 评分参考

适用场景：用户提供原工程 + 新增代码、模块接入结果、页面接入 diff，目标是评价增量开发是否自然融入既有工程。

## 一级维度与权重

| 一级维度 | 权重 | 核心关注点 |
|---|---:|---|
| 与既有工程一致性 | 20 | 命名风格、目录接入、复用既有能力 |
| 代码正确性与静态质量 | 20 | ArkTS/ArkUI 语法、类型安全、明显错误 |
| 架构与职责划分 | 15 | 新增代码边界、状态接入、能力抽取 |
| 可维护性与可读性 | 15 | 命名、复杂度、可演进性 |
| 平台规范符合度 | 15 | ArkUI 组织、ArkTS 约束、HarmonyOS 工程实践 |
| 风险控制与稳定性 | 10 | 稳定性、性能、安全边界 |
| 改动范围合理性 | 5 | 改动聚焦度、无关改动控制 |

- ArkTS/ArkUI语法与类型安全（8）

- 明显错误密度（6）

### 架构与职责划分（15）
- 新增代码职责边界（5）
- 状态接入合理性（5）
- 组件/能力抽取合理性（5）

### 可维护性与可读性（15）
- 命名表达清晰度（5）
- 复杂度控制（5）
- 可演进性（5）

### 平台规范符合度（15）

## 细化评分表

说明：二级指标按下表离散档位给分，重点看“是否自然融入原工程”；若 `reference_scope` 不足以支撑判断，应降低置信度并进入人工复核。

### 与既有工程一致性

| 二级指标 | 分值 | 给分标准 |
|---|---:|---|
| 命名与风格一致性 | 7 | 新增代码在命名、注释习惯、文件组织、格式风格上与原工程高度一致，几乎看不出“外来痕迹”。 |
| 命名与风格一致性 | 5 | 主体风格一致，少量局部命名或书写习惯偏离，但不影响整体融入感。 |
| 命名与风格一致性 | 3 | 能接入原工程，但新增代码有较明显自成体系的风格。 |
| 命名与风格一致性 | 1 | 风格差异明显，review 时可以一眼区分出新增部分与原代码。 |
| 命名与风格一致性 | 0 | 基本未对齐既有风格，新增代码与原工程割裂。 |
| 目录/模块接入一致性 | 7 | 新增文件落位、模块接入方式、引用方向和组织层级与原工程模式一致，没有新建不必要旁路结构。 |
| 目录/模块接入一致性 | 5 | 接入路径总体合理，少量文件落位或模块切分略偏，但仍可接受。 |
| 目录/模块接入一致性 | 3 | 功能勉强接入成功，但目录或模块接法与原工程习惯不完全匹配。 |
| 目录/模块接入一致性 | 1 | 接入方式明显偏离原工程模式，后续维护者需要重新适应。 |
| 目录/模块接入一致性 | 0 | 接入策略明显错误，破坏原有结构秩序。 |
| 复用既有能力程度 | 6 | 优先复用原工程已有组件、状态、工具、服务和封装能力，只在必要处新增实现。 |
| 复用既有能力程度 | 4 | 大部分已复用，但仍有局部重复实现或没有充分利用已有公共能力。 |
| 复用既有能力程度 | 2 | 明显存在“能复用却不复用”的情况，新增实现和既有能力并存。 |
| 复用既有能力程度 | 0 | 大量重复造轮子，基本无视原工程积累。 |

### 代码正确性与静态质量

| 二级指标 | 分值 | 给分标准 |
|---|---:|---|
| ArkTS/ArkUI语法与类型安全 | 8 | 增量代码无明显 ArkTS/ArkUI 语法问题，类型接入稳定，与原工程接口和状态定义衔接自然。 |
| ArkTS/ArkUI语法与类型安全 | 6 | 存在少量局部类型宽松或边界处理不够严谨，但不影响整体接入稳定性。 |
| ArkTS/ArkUI语法与类型安全 | 4 | 有多处类型或语法可疑点，接入质量需要补修。 |
| ArkTS/ArkUI语法与类型安全 | 2 | 明显存在较多类型问题、未定义引用、接口对不齐或 ArkTS 约束冲突。 |
| ArkTS/ArkUI语法与类型安全 | 0 | 高密度静态错误，增量代码本身缺乏基本可接手性。 |
| 静态坏味道控制 | 6 | 新增代码在重复逻辑、死代码、魔法值、冗余包装和过长方法方面控制良好。 |
| 静态坏味道控制 | 4 | 有少量坏味道，但局部且容易清理。 |
| 静态坏味道控制 | 2 | 坏味道较多，已经拉低增量代码质量。 |
| 静态坏味道控制 | 0 | 坏味道密集，说明增量开发质量不稳定。 |
| 明显错误密度 | 6 | 几乎无明显错误，局部问题不形成模式。 |
| 明显错误密度 | 4 | 有少量单点错误，但容易定位和修复。 |
| 明显错误密度 | 2 | 多个修改点存在明显错误，可信度下降。 |
| 明显错误密度 | 0 | 错误呈片状分布，增量代码整体不可靠。 |

### 架构与职责划分

| 二级指标 | 分值 | 给分标准 |
|---|---:|---|
| 新增代码职责边界 | 5 | 新增页面、组件、服务、状态、工具的职责与原工程边界兼容，没有把新需求硬塞进不适合的层。 |
| 新增代码职责边界 | 4 | 主要边界合理，少量实现位置不够理想。 |
| 新增代码职责边界 | 3 | 基本能工作，但边界开始松动，需要靠上下文理解职责归属。 |
| 新增代码职责边界 | 2 | 多处边界不清，新增逻辑对原结构造成污染。 |

| 状态接入合理性 | 5 | 新增状态接入遵循原工程状态管理模式，状态来源、更新链路和消费位置清晰。 |
| 状态接入合理性 | 4 | 基本合理，个别状态落位或更新路径不够优。 |

| 组件/能力抽取合理性 | 2 | 抽取不足或过度抽取较明显，增加理解成本。 |

## 本类型常见风险
- 只看新增文件，不对照原工程模式就下结论。
- 忽略无关改动、顺手重构、样式噪音。
- 对“复用不足”和“团队允许局部实现”判断不清。
- reference_scope 不足时仍给出高置信度结论。

## 输出时应强调
- 新增代码是否“融入原工程”。
- 改动是否聚焦，是否存在无关改动。
- 复用既有能力是否充分。
- 证据不足时要明确加入人工复核项。
