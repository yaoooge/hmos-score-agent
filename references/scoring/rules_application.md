# 规则层应用说明

本文件说明在 HarmonyOS NEXT / ArkTS / ArkUI 生成代码评分中，如何把规则包判定结果应用到 rubric 评分、硬门槛、人工复核和报告输出中。

## 文档定位

本文件是评分参考说明，不是规则定义源。

- 规则定义以工程内维护的规则包和导出的规则包文件为准。
- rubric 主体以 `rubric.yaml`、`full_generation_rubric.md`、`continuation_rubric.md`、`bug_fix_rubric.md` 为准。
- 本文件只说明“规则命中以后如何影响评分与报告”。
- 评审时不得引用当前规则包中不存在的规则编号，也不得沿用旧版本中已删除或替换的规则编号。

## 基本原则

最终得分不是“rubric 分 + 独立规则分”，而是：

> 基础 rubric 得分 + 规则层修正结果，并受硬门槛、总分上限和人工复核规则约束。

规则层只修正既有评分项，不单独形成第二套总分。

规则处理优先级如下：

1. HarmonyOS NEXT / ArkTS / ArkUI 平台强约束
2. 必须满足类规则
3. 禁用模式类规则
4. 建议类规则
5. 性能实践类建议与禁用模式

当规则结论与 rubric 结论冲突时，应以可验证证据为准；证据不足时降低置信度并进入人工复核，不允许强行判定。

## 三类任务的应用重点

### 完整生成

完整生成场景评价的是新工程产物是否具备可接手性。
完整生成中，成体系的必须规则违规或禁用模式命中，通常比单点风格问题更重要。若违规集中出现在类型系统、模块系统、动态对象布局或平台组织方式上，应优先评估 G1、G2 或 G3。

### 续写接入

续写接入场景评价的是新增代码是否自然融入既有工程。规则层应同时看新增代码自身质量和它对原工程的影响。
续写接入中，同一条规则命中应优先映射到“与既有工程一致性”“平台规范符合度”“代码正确性与静态质量”和“改动范围合理性”等维度，而不是只按完整生成逻辑处理。

### Bug 修复

Bug 修复场景评价的是修复是否命中问题、是否控制侵入范围、是否引入新风险。
Bug 修复中，规则违规不只代表代码质量下降，还可能说明误修、过修或修复方式不可信。若改动与问题点不匹配，或修改范围严重失控，应优先评估 G4。

## 规则类别处理

### 必须满足类规则

必须满足类规则用于识别 ArkTS 语言约束、类型系统约束、模块系统约束、对象布局约束、异常约束和明确要求的编码规范。

处理原则：

- 明确不满足时，至少降低相关二级指标分数。
- 若属于编译失败级、类型系统破坏级、平台约束直接冲突或高风险控制流错误，应进入硬门槛候选。
- 报告中必须写明规则编号、规则摘要、证据、影响项和处理结果。
- 若只有可疑信号但没有足够上下文，只能标为待人工复核。

### 禁用模式类规则

禁用模式类规则用于标记高风险反模式。它们通常比一般建议项更适合进入风险项或人工复核。

处理原则：

- 一旦确认命中，必须进入规则违规标记。
- 通常应进入风险项；影响严重时进入硬门槛候选。
- 不得只写“疑似不规范”，必须说明命中的具体禁用模式。
- 若命中来自正则、片段或局部结构，仍需确认是否存在误报；误报风险较高时进入人工复核。

### 建议类规则

建议类规则用于评价迁移质量、工程风格、可维护性和性能实践。它们通常不直接触发硬门槛。

处理原则：

- 单条轻微命中一般只作为相关二级指标的修正因素。
- 同类问题集中出现时，应在主要问题中归纳为模式性缺陷。
- 不应把建议类规则机械等同于严重违规。
- 若建议类规则涉及性能敏感路径、状态传播或跨模块影响，可提高风险等级但仍需证据支撑。

## 规则到 rubric 的映射

规则命中后，应映射到当前任务 rubric 中已有二级指标。常见映射如下。

| 规则问题类型 | 主要影响指标 |
|---|---|
| 类型系统、弱类型逃逸、类型断言、索引签名、结构类型依赖 | 代码正确性与静态质量；平台规范符合度；可维护性与可读性 |
| ArkTS / ArkUI 语法或平台约束冲突 | 代码正确性与静态质量；平台规范符合度 |
| 模块系统违规、导入导出错位、跨层依赖异常 | 工程结构与组织质量；与既有工程一致性；风险控制与稳定性 |
| 动态对象布局、动态属性访问、受限动态接口 | 平台规范符合度；风险控制与稳定性；安全与边界意识 |
| 异常处理、空值、异步时序、资源释放、监听清理风险 | 风险控制与稳定性；代码正确性与静态质量 |
| 控制流复杂、嵌套过深、样板遍历、非正常结束 | 可维护性与可读性；静态坏味道控制；稳定性风险 |
| 命名、格式、注释、文件组织风格 | 可维护性与可读性；与既有工程一致性；静态坏味道控制 |
| 状态传递、全局状态、状态来源不清 | 架构与职责划分；风险控制与稳定性 |
| 性能实践问题 | 风险控制与稳定性；性能风险；代码正确性与静态质量 |
| 用例 P0 约束不满足 | 硬门槛候选；相关任务目标指标；人工复核 |

同一条规则可以影响多个指标，若一条证据已经用于解释某个核心扣分项，其他指标应只补充说明它的连带影响。

## 硬门槛识别

硬门槛用于限制自动评分的最高可信区间。触发硬门槛时，最终报告必须进入人工复核。

### G1 高密度静态错误

关注信号：

- 多条必须满足类规则同时不满足。
- 类型系统、模块系统、导入导出或未定义引用问题集中出现。
- 代码明显无法通过 ArkTS 静态检查或需要大面积补修。
- 用例 P0 约束明确不满足。

证据充分时，总分上限为 69。

### G2 明显不符合 HarmonyOS NEXT / ArkTS 基本规范

关注信号：

- 大量 TypeScript 动态能力被直接搬入 ArkTS。
- 使用弱类型、结构类型、动态对象、受限模块系统或受限标准库接口形成体系性问题。
- ArkUI 页面、组件、状态组织明显硬套其他前端框架思路。

证据充分时，总分上限为 69。

### G3 严重工程风险

关注信号：

- 动态对象布局、异常值滥用、受限动态接口、资源释放缺失等引出稳定性风险。
- 生命周期、异步时序、空值处理、权限、资源或敏感信息边界存在明显高危点。
- 性能禁用模式出现在核心链路或热点路径，并可能影响稳定性或体验。

证据充分时，总分上限为 79。

### G4 Bug 修复任务中的误修或过修

仅适用于 Bug 修复场景。

关注信号：

- 改动与问题点明显不匹配。
- 修改范围严重失控。
- 引入大量无关重构、格式噪音或旁路结构。
- 为修复引入弱类型逃逸、忽略类型检查、动态访问、模块违规或新的高风险控制流。

证据充分时，总分上限为 59。

## 人工复核触发条件

满足任一条件时，应写入人工复核：

- 证据不足，无法确认是否真实命中规则。
- 续写接入或 Bug 修复缺少足够参考上下文。
- Bug 修复无法确认是否命中根因。
- 辅助判断不可用、输出不完整或结论置信度低。
- 同时命中多条规则，但影响范围和主次关系不清晰。
- 触发任意硬门槛候选。
- 总分落在 68-71、78-81、88-91 临界带。
- rubric 结论与规则层结论明显冲突。
- 规则命中疑似误报，但自动流程无法排除。

人工复核项应说明当前判断、为什么不确定、建议人工重点检查什么。

## 证据要求

确认规则不满足时，至少需要以下一种证据：

- 文件路径和明确代码片段。
- 补丁位置和变更前后关系。
- 工程结构事实，如目录、模块、资源或配置关系。
- 用例约束与实现之间的明确不一致。
- 多处重复出现的同类模式。

证据不足时：

- 不得写成已确认违规。
- 不得用“可能”“看起来像”直接扣成严重问题。
- 应写为待人工复核，并说明缺少哪类证据。

对于正则或文本模式命中的规则，应警惕误报。只有当片段语义与规则摘要一致时，才能写为明确不满足。

## 报告写法

当发现规则命中时，报告应包含以下内容：

- 规则来源：平台强约束、必须满足类规则、建议类规则或禁用模式类规则。
- 规则编号：使用当前规则包中的编号。
- 规则摘要：直接使用或忠实转述当前规则包中的规则内容。
- 影响项：说明映射到哪些 rubric 二级指标。
- 处理结果：降分、风险提示、硬门槛候选或人工复核。
- 证据：文件路径、代码片段、补丁位置或结构事实。



## 当前优先复核清单

评审时可优先检查以下问题：

- 是否存在弱类型逃逸、类型检查绕过或不受支持的 TypeScript 类型能力。
- 是否存在动态对象布局、动态属性访问或受限动态标准库接口。
- 是否存在模块系统违规或跨语言、跨文件依赖方向异常。
- 是否存在异常处理、`finally`、生命周期、异步时序或资源释放风险。
- 是否存在性能禁用模式，尤其是核心链路中的联合类型数组、稀疏数组、数值混用或热点路径抛异常。
- 续写接入是否复用原工程能力，是否保持目录、命名、状态和组件抽取方式一致。
- Bug 修复是否命中问题点，是否存在误修、过修或无关重构。
- 用例 P0 约束是否有明确满足证据。
