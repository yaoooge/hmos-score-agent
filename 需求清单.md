# 评分服务需求清单（原始需求归档）

## 用户原始目标

在当前目录下，使用 LangGraph 的 TypeScript 框架实现鸿蒙代码评分服务。

## 原始需求逐条归档

1. 这是一个给鸿蒙代码评分的 workflow，输入代码由其他 agent 根据 prompt 智能生成。

2. 使用 agent 串联各个步骤，需要支持独立部署到云服务器上。

3. 初始输入参考 `init-input` 目录。未来每个用例都包含该目录下的所有内容：
   - 原始工程
   - 提供给代码生成 AI 的 input（prompt）
   - 生成后完整工程
   - 前后差异 patch 文件
   - 后续会通过网络链接下载，需要预留对应接口适配

4. 工程本地运行时，步骤中提到的落盘都放在指定目录下：
   - 一条用例一个文件夹
   - 目录加入 `.gitignore`
   - 仅本地暂存
   - 后续部署到云端执行时，落盘文件需要上传，需要预留上传操作

5. 评分步骤如下：

   1) 任务理解  
   - 根据原始 prompt 获取显式约束（任务类型/行业/场景/目标等）  
   - 根据原工程获取上下文约束（结构/模块/实现方式）  
   - 根据代码 patch 获取隐式约束（修改范围/改动类型/侵入程度等）  
   - 最好做成可配置（yaml 或其他形式），便于后续增加输入和理解内容  
   - 将约束结论落盘

   2) 输入分类  
   - 根据约束结论将用例分为 `full_generation`、`bug_fix`、`continuation`  
   - 按不同类型检索不同 rubric  
   - 三种类型 rubric 参考：
     - `/Users/guoyutong/.claude/skills/harmonyos-gen-code-evaluator/references/full_generation_rubric.md`
     - `/Users/guoyutong/.claude/skills/harmonyos-gen-code-evaluator/references/bug_fix_rubric.md`
     - `/Users/guoyutong/.claude/skills/harmonyos-gen-code-evaluator/references/continuation_rubric.md`
   - 将前面获取信息组装成 prompt，交给评分 agent 处理

   3) 代码解析与特征提取  
   - 根据工程信息和 prompt 进行代码解析，获取：
     - 基础特征：状态管理类型/路由等
     - 结构：工程架构/代码分层等
     - 语义：命名/关键字
     - 变更：最小改动/无关侵入
   - 需预留后续扩展（如三方库、性能等特征）

   4) 代码评测  
   - 根据 rubric 和静态规则进行评测判断  
   - 静态规则参考：
     - `/Users/guoyutong/.claude/skills/harmonyos-gen-code-evaluator/references/arkts_internal_rules.yaml`
   - 逐条输出静态规则校验结果并落盘

   5) 评分编排  
   - 根据 rubric 进行加权平均计算  
   - 识别硬门槛  
   - 判断人工复核项  
   - 计算最终得分  
   - 编排可参考：
     - `/Users/guoyutong/.claude/skills/harmonyos-gen-code-evaluator/SKILL.md`

   6) 输出结果  
   - 输出固定 schema 的 `result.json` 与可视化 `html` 报告  
   - schema 参考：
     - `/Users/guoyutong/.claude/skills/harmonyos-gen-code-evaluator/references/report_result_schema.json`
   - 需要将 `json` 上传云端管理台  
   - `html` 报告落盘

## 已确认实现偏好（会话内确认）

- LLM 接口：`兼容 chat completions 的模型服务`（支持自定义 `baseURL + API Key`）
- 上传接口：`HTTP REST`（默认上传 `result.json`）
