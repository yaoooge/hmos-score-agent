## 2. 与需求直接相关的关键约束

### 2.1 任务类型

评分任务必须先分类为三种之一：
- `full_generation`
- `continuation`
- `bug_fix`

分类依据来自输入形态（原工程、生成工程、patch、任务描述）与约束抽取结果。

### 2.2 评分边界

评分关注：
- 工程组织质量
- 代码静态质量
- 架构与职责划分
- 可维护性与可读性
- HarmonyOS / ArkTS / ArkUI 规范符合度
- 风险控制
- 与既有工程一致性
- 改动精准度与最小侵入性

默认不作为主评分依据：
- 功能是否跑通
- UI 美观度
- 产品设计质量
- 测试验收结果
- 编译通过率

### 2.3 规则处理优先级

按下列顺序应用：
1. 平台强约束（HarmonyOS / ArkTS / ArkUI）
2. `must_rules`
3. `forbidden_patterns`
4. `should_rules`

## 3. Rubric 结构要点

`rubric.yaml` 提供：
- 三类任务的一级维度与二级指标权重
- 硬门槛定义（G1/G2/G3/G4）
- 人工复核触发条件
- 规则接入模式（`additive_modifier` / `gating_only` / `review_only`）

其中：
- `full_generation` 更关注工程可接手性与结构完整性
- `continuation` 更关注与既有工程一致性、改动聚焦度
- `bug_fix` 更关注命中问题点、最小侵入与回归风险

## 4. 规则集覆盖要求

`src/rules/packs/` 中注册的规则必须全量审计，主要类别包括：
- `must_rules`
- `should_rules`
- `forbidden_patterns`

逐条审计结果必须输出为：
- `满足`
- `不满足`
- `不涉及`

并保持规则顺序与源 YAML 一致。

## 5. 输出 schema 关键约束

`result.json` 必须严格符合 `report_result_schema.json`：
- 不允许新增字段
- 不允许改变字段语义
- 顶层 required 字段必须齐全

特别注意：
- `rule_audit_results` 每条仅允许 4 个字段：
  - `rule_id`
  - `rule_source`（`must_rule|should_rule|forbidden_pattern`）
  - `result`（`满足|不满足|不涉及`）
  - `conclusion`
- 若规则判为 `不满足`，需同步写入 `rule_violations`

## 6. 硬门槛与人工复核

硬门槛（来自 rubric）：
- G1 高密度静态错误（总分上限 69）
- G2 明显不符合 ArkTS 基本规范（总分上限 69）
- G3 严重工程风险（总分上限 79）
- G4 bug_fix 误修或过修（总分上限 59）

人工复核典型触发：
- 任一指标 `confidence = low`
- continuation / bug_fix 缺少对照上下文
- 触发硬门槛
- 总分处于临界带（68-71、78-81、88-91）

## 7. 对当前工程实现的落地映射

与本仓库设计文档 `评分服务设计文档.md` 对齐的实现检查项：
- 任务理解节点支持显式/上下文/隐式约束抽取并落盘
- 输入分类节点按三类任务选择对应 rubric
- 规则评测节点全量遍历规则并输出逐条台账
- 评分编排节点执行加权、硬门槛、人工复核判定
- 报告节点输出固定 schema `result.json` 与 `report.html`
- 上传节点按 HTTP REST 上传 `result.json`

## 8. 易漏项清单（开发时重点核对）

- 是否对 `src/rules/packs/` 全量注册规则都给出结论（不是只输出命中项）
- 是否将 `rule_audit_results` 与 `rule_violations` 分开建模
- 是否按任务类型切换到正确的评分维度权重
- 是否在证据不足时降低 `confidence` 并写人工复核项
- 是否保证 `result.json` 不新增字段且 required 完整
