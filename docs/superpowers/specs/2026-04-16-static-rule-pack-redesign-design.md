# 静态规则包重构与未接入判定器设计

## 1. 背景

当前工程已经具备基础的静态规则审计能力，但实现方式仍然停留在单文件映射阶段：

- `src/rules/ruleMapping.ts` 只维护少量规则的文本匹配映射
- `src/rules/textRuleEvaluator.ts` 在规则未命中映射时，直接返回 `不涉及`
- `src/rules/ruleEngine.ts` 按 `arkts_internal_rules.yaml` 的顺序扫描规则，但缺少“规则已判定 / 未接入判定器 / 需 agent 辅助”的明确分层

这会带来三个直接问题：

1. 规则覆盖率极低，当前 63 条规则中只有极少数能被静态判定。
2. “未接入判定器”与“确实不涉及”被混在一起，导致结果语义失真。
3. 后续扩展新规则集时，继续堆叠在单一 `ruleMapping.ts` 中会快速失控，不利于维护、测试和扩展。

同时，上一份设计文档 [`2026-04-16-agent-assisted-rule-evaluation-design.md`](/Users/guoyutong/MyWorkSpace/hmos-score-agent/docs/superpowers/specs/2026-04-16-agent-assisted-rule-evaluation-design.md) 已经明确了总体方向：

- 规则引擎优先
- agent 只辅助判定本地无法稳定确定的规则
- 最终评分只消费本地归一化后的统一规则结果

因此，这一轮需要补齐静态规则引擎的结构性设计：先把规则包组织方式、判定状态模型和与 agent 的衔接边界定义清楚，再进入 TDD 开发。

## 2. 目标

本轮设计目标如下：

1. 将当前静态规则能力从“单文件平铺映射”重构为“按规则包组织”的结构。
2. 以当前规则集《从 TypeScript 到 ArkTS 的适配规则》与《ArkTS 编程规范》融合包为第一批落地对象。
3. 按规则来源维度组织文件，而不是按更细的语法子类继续过度拆分。
4. 对 63 条规则逐条建立归档与分类：
   - 能静态判定的规则，本轮接入本地判定器
   - 暂时不能静态判定的规则，明确标记为 `未接入判定器`
5. 让 `未接入判定器` 成为工作流内部状态，并作为下一轮 agent 辅助判定的唯一主要输入来源。
6. 保持最终评分、报告和对外结果的语义稳定，不把内部状态直接暴露到最终 `mergedRuleAuditResults` 中。

## 3. 非目标

本轮明确不做以下事项：

- 不在这一轮引入 AST-first 规则引擎
- 不在这一轮引入人工复审知识库回流
- 不在这一轮重写评分算法
- 不在这一轮修改 `result.json` 的对外 schema 字段名
- 不为了规则拆分而过度增加目录层级
- 不要求所有规则都达到 AST 级精度，只要求“能稳定静态判定的优先落地”

## 4. 设计原则

### 4.1 规则包优先于规则类型细分

目录先按“规则集”拆包，再在包内按 `must / forbidden / should` 划分。当前所有规则都属于同一个 ArkTS 规则包，因此不再继续按 `types / functions / modules / naming` 等维度分目录，避免文件结构过度碎片化。

### 4.2 明确区分“不涉及”和“未接入判定器”

这两个状态语义完全不同：

- `不涉及`：规则已被判定，当前输入不触发该规则
- `未接入判定器`：当前版本还没有为该规则提供静态判定能力

静态层必须严格区分二者，不能再用 `不涉及` 去掩盖能力缺口。

### 4.3 内部状态可以更细，但对外结果必须稳定

工作流内部允许出现 `未接入判定器`，因为这是真实状态，也决定后续是否进入 agent 辅助判定。

但最终给评分和报告消费的结果，仍然只保留：

- `满足`
- `不满足`
- `不涉及`
- `待人工复核`

也就是说，`未接入判定器` 只存在于静态层和 agent 前置链路，不直接出现在最终合并结果中。

### 4.4 规则迁移按“先能判，再补全”执行

本轮不是简单把规则文案抄进新目录，而是要对 63 条规则逐条做一轮能力审视：

- 可以稳定用文本/文件结构/项目特征判定的，直接实现
- 目前不适合稳定静态判定的，先归档为 `未接入判定器`

这样迁移后的规则引擎是“真实能力清单”，不是“看起来全量接入”的空壳。

### 4.5 与 agent 的边界保持单向清晰

静态规则层负责：

- 全量规则枚举
- 能判定的先判定
- 不能判定的明确标记为 `未接入判定器`

agent 层负责：

- 只接手 `未接入判定器` 的候选规则
- 输出辅助结论
- 由本地代码映射回最终统一结果

不能让 agent 反向覆盖静态已经确定的规则结论。

## 5. 目录与模块设计

### 5.1 目标目录结构

```text
src/rules/
  engine/
    ruleEngine.ts
    rulePackRegistry.ts
    ruleTypes.ts
  evaluators/
    textPatternEvaluator.ts
    projectStructureEvaluator.ts
    shared.ts
  packs/
    arkts-language/
      must.ts
      forbidden.ts
      should.ts
  evidence/
    evidenceCollector.ts
    evidenceSelectors.ts
```

如果当前重构成本需要控制，允许保留现有部分文件名，但目标职责边界必须与上面的结构一致。尤其是：

- 规则定义与规则引擎分离
- 判定器实现与规则声明分离
- 证据采集逻辑与规则执行逻辑分离

### 5.2 每层职责

#### `packs/*`

负责声明规则，不负责执行。每条规则至少描述：

- `rule_id`
- `rule_source`
- `summary`
- `detector_kind`
- `detector_config`
- `fallback_policy`

#### `evaluators/*`

负责执行具体判定逻辑，例如：

- 正则文本命中
- 特定文件是否存在
- import / API / 关键字扫描
- 基于收集证据的结构化条件判断

#### `engine/*`

负责：

- 加载规则包
- 按顺序执行规则
- 统一返回静态判定结果
- 生成后续 agent 候选列表

#### `evidence/*`

负责：

- 收集 `workspace/original/patch` 证据
- 提供按后缀、路径、改动文件、代码片段筛选能力
- 为静态规则和 agent 共享证据索引

## 6. 规则模型设计

### 6.1 新增内部判定状态

建议在静态规则层引入内部结果枚举：

- `满足`
- `不满足`
- `不涉及`
- `未接入判定器`

其中：

- `满足 / 不满足 / 不涉及` 表示静态层已经完成判定
- `未接入判定器` 表示当前规则还没有可用静态判定逻辑

### 6.2 对外结果与内部结果分离

建议区分两个结构：

#### `StaticRuleAuditResult`

供静态层和 agent 前置链路使用，允许包含 `未接入判定器`。

#### `RuleAuditResult`

供最终评分、报告与落盘使用，不新增对外枚举值，继续保持：

- `满足`
- `不满足`
- `不涉及`
- `待人工复核`

### 6.3 状态流转

静态层到最终结果的流转规则如下：

```text
静态判定 = 满足            -> 最终结果 = 满足
静态判定 = 不满足          -> 最终结果 = 不满足
静态判定 = 不涉及          -> 最终结果 = 不涉及
静态判定 = 未接入判定器    -> 进入 agent 辅助链路

agent 结论 = 满足          -> 最终结果 = 满足
agent 结论 = 不满足        -> 最终结果 = 不满足
agent 结论 = 不确定/证据不足 -> 最终结果 = 待人工复核
agent 不可用/输出非法       -> 最终结果 = 待人工复核
```

## 7. 规则包拆分方案

### 7.1 当前包命名

当前规则统一归入：

- `arkts-language`

这一个包同时承接两类来源：

- 《从 TypeScript 到 ArkTS 的适配规则》
- 《ArkTS 编程规范》

这里不按来源再拆两个包，因为当前融合规则文件已经以同一优先级整体使用，继续拆来源会增加运行时合并复杂度，但不会带来等价收益。

### 7.2 包内文件划分

固定三文件：

- `src/rules/packs/arkts-language/must.ts`
- `src/rules/packs/arkts-language/forbidden.ts`
- `src/rules/packs/arkts-language/should.ts`

划分依据不是技术域，而是评分规则的消费语义：

- `must` 通常直接影响硬性规范与静态质量
- `forbidden` 表示高风险禁用模式
- `should` 表示建议型或需进一步上下文判断的规则

### 7.3 后续扩展方式

后续新规则集按相同结构扩展，例如：

```text
src/rules/packs/performance/
  must.ts
  forbidden.ts
  should.ts

src/rules/packs/third-party-library/
  must.ts
  forbidden.ts
  should.ts
```

这样可以保证：

- 每个规则集边界清晰
- 包间互不干扰
- 新规则集接入成本可控

## 8. 63 条规则迁移策略

### 8.1 迁移步骤

对 `arkts_internal_rules.yaml` 中的 63 条规则逐条执行以下动作：

1. 记录规则归属到 `must / should / forbidden`
2. 判断是否可以通过当前版本静态能力稳定判定
3. 若可静态判定，分配到对应的 evaluator
4. 若暂不可静态判定，标记为 `未接入判定器`
5. 为每条规则补充中文结论模板，避免后续报告混入英文描述

### 8.2 “可静态判定”的标准

满足以下任一条件即可优先接入静态判定：

- 可用明确关键字或语法片段命中
- 可通过 import / API 名称检测
- 可通过文件后缀、路径、目录结构判断
- 可通过改动文件范围和已收集证据稳定判定
- 不需要复杂跨文件语义推理

### 8.3 “未接入判定器”的判定标准

符合以下特征之一的规则，先标记为 `未接入判定器`：

- 需要较强语义理解
- 需要跨多个文件的高阶关系推断
- 仅靠文本匹配误报率过高
- 当前没有 AST / 类型信息支撑而无法稳定判断

## 9. 与工作流的衔接调整

### 9.1 `ruleAuditNode`

`ruleAuditNode` 需要从“直接产出最终 `ruleAuditResults`”调整为同时产出三类数据：

- `staticRuleAuditResults`
- `deterministicRuleResults`
- `assistedRuleCandidates`

其中：

- `deterministicRuleResults` = 所有静态已完成判定的规则
- `assistedRuleCandidates` = 所有 `未接入判定器` 的规则

### 9.2 `agentPromptBuilderNode`

候选规则来源需要从“弱规则/不确定规则”更新为：

- 所有静态结果为 `未接入判定器` 的规则

这使 agent 的输入边界更稳定，也更符合“规则引擎优先”的设计原则。

### 9.3 `ruleMergeNode`

合并阶段需要遵守如下规则：

- 静态已确定的结果不可被 agent 覆盖
- 仅合并静态为 `未接入判定器` 的规则
- agent 若给不出确定结论，统一落为 `待人工复核`

## 10. 日志与落盘设计

为便于调试与规则能力评估，建议新增或保留以下中间产物：

- `intermediate/static-rule-audit.json`
  记录包含 `未接入判定器` 的静态原始结果
- `intermediate/rule-audit.json`
  记录当前阶段最终用于评分的规则结果
- `intermediate/rule-candidates-for-agent.json`
  记录进入 agent 的候选规则
- `intermediate/rule-audit-merged.json`
  记录 agent 合并后的最终规则结果

同时在 `logs/run.log` 中增加关键日志：

- 静态规则执行开始与结束
- 静态可判定规则数
- `未接入判定器` 规则数
- 进入 agent 的候选规则数
- agent 合并后转为 `待人工复核` 的规则数

所有描述型文案继续保持中文。

## 11. 测试策略

本轮开发必须遵循 TDD，并至少覆盖以下测试层次。

### 11.1 规则定义测试

验证：

- 所有 63 条规则都已在对应规则包中注册
- 不存在缺失、重复或错误归类
- `must / should / forbidden` 数量与源规则文件一致

### 11.2 evaluator 单元测试

对每种静态判定器分别编写：

- 命中测试
- 未命中测试
- 文件后缀过滤测试
- 误报控制测试

### 11.3 规则引擎集成测试

验证：

- 规则执行顺序与源规则一致
- `未接入判定器` 不再被错误映射成 `不涉及`
- 候选规则列表只包含 `未接入判定器`

### 11.4 工作流测试

验证：

- `ruleAuditNode` 正确拆出静态结果和 agent 候选
- `ruleMergeNode` 正确将 agent 结果映射回统一结果
- agent 不可用时，候选规则统一回退为 `待人工复核`

## 12. 实施顺序建议

### 第一阶段：模型和结构重构

- 建立规则包目录结构
- 建立内部静态规则结果类型
- 引入注册表与执行入口

### 第二阶段：迁移现有已支持规则

- 先迁移当前已支持的少量规则
- 保持原有测试持续通过

### 第三阶段：逐条补齐 63 条规则

- 可静态判定的规则逐条接入
- 暂不可静态判定的规则标记为 `未接入判定器`

### 第四阶段：打通 agent 候选链路

- 将 `未接入判定器` 规则接入 agent prompt 构建
- 调整合并逻辑和日志

## 13. 风险与取舍

### 13.1 风险：文本匹配规则误报

快速落地版本仍会以文本和结构判定为主，部分规则可能存在误报或漏报。

应对策略：

- 只把“稳定可判”的规则纳入静态判定
- 不稳定规则直接标记为 `未接入判定器`

### 13.2 风险：一次性迁移 63 条规则工作量较大

逐条审阅和落地 63 条规则会带来较高实现成本。

应对策略：

- 先完成规则包结构与测试骨架
- 再按规则逐条补齐，确保每条规则状态真实

### 13.3 取舍：暂不引入 AST

不引入 AST 会牺牲部分准确率，但能显著降低本轮实现周期，更符合“先快速落地一个版本”的目标。

## 14. 结论

本轮推荐采用“规则包注册表 + 包内 `must / forbidden / should` 三文件 + 内部 `未接入判定器` 状态”的方案。

该方案兼顾了四个目标：

1. 能在当前工程基础上快速落地
2. 能真实表达静态规则引擎当前能力边界
3. 能自然衔接下一轮 agent 辅助判定
4. 能为后续性能规范、三方库规范等新规则集扩展保留清晰结构

在此方案下，静态层负责“尽可能多地稳定判定”，agent 层负责“补位未接入判定器”，最终评分层继续消费稳定的统一结果类型。这是当前阶段最稳妥、可演进且符合工程现实的落地路径。
