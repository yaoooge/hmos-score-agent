# 首版本去兼容化清理设计

## 1. 背景

当前工程仍保留了多处“兼容旧版本/历史口径”的实现和表述，但该项目截至目前仍处于首版本阶段，并不存在需要兼容的历史发布版本。因此，这些兼容性逻辑会带来两个直接问题：

- 运行时语义变得含混，真实的一手状态与“兼容输出”状态并存
- 后续开发会误以为某些历史格式、旧字段或降级分支是必须长期维护的契约

本轮目标不是增加新能力，而是把首版本的唯一真实行为收口为一套明确语义，并删除所有“为了兼容历史而存在”的运行时、状态、测试和文档内容。

## 2. 目标

本轮清理目标如下：

1. 删除运行时中所有“兼容旧版本”的分支和命名。
2. 删除静态规则链路中所有“旧结果口径映射”。
3. 保证工作流各节点只消费首版本的真实状态，而不是兼容层产物。
4. 删除测试、注释、文档中与“旧版本兼容”相关的表述和断言。
5. 保持当前首版本的真实功能闭环可用：
   - 静态规则识别
   - agent 辅助判定
   - 规则结果合并
   - 评分与报告输出
6. 在中间产物和输入落盘文件中，移除 `prompt.txt` / `original-prompt.txt` 这类首版本不再需要保留的快照文件。

## 3. 非目标

本轮明确不做以下事项：

- 不调整评分算法
- 不修改对外 `result.json` schema
- 不扩展新的规则包或判定器能力
- 不重构无关命名或目录结构
- 不引入新的兼容层来替代旧兼容层
- 不改变评分主链的执行顺序

## 4. 设计原则

### 4.1 首版本只保留一套真实语义

任何“兼容旧版本”的中间状态、旧字段映射、降级分支都应删除。系统内部和测试都只围绕当前版本的真实契约构建。

### 4.2 兼容命名不是中性命名

凡是名称中包含 `Compatible`、`legacy` 或明确暗示“旧版本适配”的类型、注释和测试名，都应被替换。首版本实现应使用直接、稳定的命名，而不是带迁移背景的命名。

### 4.3 状态流转必须单向清晰

规则工作流中：

- 静态层产生 `staticRuleAuditResults`
- 可判定项进入 `deterministicRuleResults`
- 未接入判定器项进入 `assistedRuleCandidates`
- 最终统一结果只由合并节点输出

不再保留任何“为了兼容旧消费方而额外生成一份旧口径结果”的逻辑。

评分主链顺序保持不变：

1. 静态规则审计
2. agent 辅助判定
3. 规则结果合并
4. 评分与报告

本轮只清理兼容层，不改变这条主链。

### 4.4 测试和文档必须与运行时一致

如果运行时已经不支持历史兼容语义，那么测试和文档也不能继续保留“兼容旧版本”的表述，否则会误导后续开发。

## 5. 清理范围

### 5.1 Agent Client

当前问题：

- client 类型名仍包含 `Compatible`
- 请求逻辑仍保留“首次带 `response_format`，失败后重试无 `response_format`”的降级路径

清理方案：

- 将 `CompatibleChatModelClient` 重命名为首版本直接命名
- 将 `CompatibleChatModelClientOptions` 改为直接命名
- 删除 `shouldRetryWithoutResponseFormat()` 及其对应的二次请求逻辑
- 保留单次请求路径，接口不接受当前协议时直接失败

影响：

- 首版本运行时只支持当前定义的请求格式
- 测试不再验证兼容接口降级行为，而只验证当前协议请求与响应解析

### 5.2 静态规则引擎

当前问题：

- `ruleEngine` 内部仍生成 `legacyRuleAuditResults`
- `未接入判定器` 会被映射成 `不涉及` 再写入 `ruleAuditResults`

这实质上是为了旧消费口径保留的一层兼容输出。

清理方案：

- 删除 `legacyRuleAuditResults`
- 删除“`未接入判定器` -> `不涉及`”的兼容映射
- `ruleEngine` 输出只保留真实分层结果：
  - `staticRuleAuditResults`
  - `deterministicRuleResults`
  - `assistedRuleCandidates`
  - `ruleViolations`
  - `ruleEvidenceIndex`
  - `evidenceSummary`

如果工作流中仍需要 `ruleAuditResults` 字段，则其定义必须改为首版本真实语义，而不是兼容旧消费方的伪装结果。

### 5.3 规则合并与下游节点

当前问题：

- `ruleMergeNode` 在无候选规则时直接回传 `state.ruleAuditResults`
- 这使工作流继续依赖旧口径字段

清理方案：

- `ruleMergeNode` 在无候选规则时直接基于 `deterministicRuleResults` 生成最终结果
- 所有下游节点统一消费：
  - 有 merge 结果时用 `mergedRuleAuditResults`
  - 无 merge 结果时用 `deterministicRuleResults`

结果：

- `ruleAuditResults` 不再承担兼容旧消费方的职责
- 全流程只保留一条真实数据路径

### 5.4 测试

需要清理的测试类型：

- 带有 `Compatible...` 命名的测试
- 验证兼容接口降级重试的测试
- 验证“旧口径结果映射仍存在”的测试
- 注释或断言中带有 `legacy`、`compatible`、`旧版本兼容` 的内容

替代方式：

- 只验证首版本当前协议
- 只验证单次请求行为
- 只验证新的真实状态流转

### 5.5 文档与注释

需要清理：

- 注释中“兼容 chat completions”“旧口径”“兼容旧版本”等描述
- 当日新增设计文档、计划文档和测试说明中保留的兼容性措辞

保留原则：

- 可以描述“当前协议”或“当前版本”
- 不再描述“不存在的历史兼容责任”

### 5.6 落盘文件

当前问题：

- 工作流仍会在输入目录落盘 `prompt.txt` / `original-prompt.txt` 一类快照文件
- 这些文件不属于“兼容旧版本”本身，但会继续固化多余的历史输入快照路径，不符合首版本收口后的最小落盘原则

清理方案：

- 在 `persistAndUploadNode` 中停止写入 `inputs/original-prompt.txt`
- 相关测试、文档、说明同步更新，不再断言或描述该文件存在
- 保留当前实际需要的落盘内容，例如：
  - `inputs/agent-prompt.txt`
  - `inputs/agent-prompt-payload.json`
  - 规则审计和合并结果
  - `outputs/result.json`
  - `outputs/report.html`

结果：

- 落盘文件只保留当前工作流真实需要的输入与产物
- 不再额外保存 prompt 快照文件

## 6. 最终状态模型

### 6.1 静态规则层

静态层允许：

- `满足`
- `不满足`
- `不涉及`
- `未接入判定器`

这部分仅存在于静态规则执行链路。

### 6.2 最终规则结果层

最终合并结果只允许：

- `满足`
- `不满足`
- `不涉及`
- `待人工复核`

生成规则：

- 确定性结果直接进入最终结果
- `未接入判定器` 不再伪装成 `不涉及`
- 需要 agent 的候选项由 merge 节点根据 agent 输出映射成最终结果
- agent 不可用或输出非法时，对应候选项进入 `待人工复核`

## 7. 实施顺序

1. 先写失败测试，锁定以下行为：
   - client 不再做降级重试
   - workflow 不再依赖兼容旧口径结果
   - 文本与命名中不再保留 `Compatible` / `legacy` 语义
   - `persistAndUploadNode` 不再写入 `prompt.txt` / `original-prompt.txt`
2. 删除 agent client 兼容分支并调整命名
3. 删除 `ruleEngine` 中的旧口径映射
4. 调整 `ruleMergeNode` 和下游节点消费真实状态，但保持“静态 -> agent -> 合并”的主链顺序不变
5. 删除 prompt 落盘文件并更新相关测试
6. 更新测试、注释和文档
7. 跑工作流相关测试集验证无回归

## 8. 风险与控制

主要风险：

- 删除 `ruleAuditResults` 的兼容职责后，部分下游节点可能仍依赖它
- 调整 client 命名后，相关测试和导入需要同步更新

控制方式：

- 用失败测试先锁住真实消费路径
- 先改状态消费，再删兼容输出
- 用工作流级测试验证：
  - 规则审计
  - agent 辅助判定
  - 合并结果
  - 评分与报告

## 9. 验收标准

满足以下条件视为完成：

1. `src` 和 `tests` 中不再存在为旧版本兼容保留的运行时分支。
2. `Compatible`、`legacy` 等历史兼容命名不再出现在首版本运行时代码中。
3. `ruleEngine` 不再生成旧口径映射结果。
4. 工作流各节点只消费首版本真实状态，且评分主链仍保持“静态 -> agent -> 合并 -> 评分”。
5. 落盘文件中不再生成 `prompt.txt` / `original-prompt.txt`。
6. 相关测试全部通过，且测试描述不再保留“兼容旧版本”语义。
