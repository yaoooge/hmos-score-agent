# 评分一致性分析管理页面设计

## 背景

当前项目已经在 `web/` 下提供 Vue/Vite 管理台，后端由评分服务 API 支撑。现有页面覆盖评测任务、用例报表、人工评分差异分析、负向结果分析、风险项分析和一多适配分析。

远端评分通过以下接口触发：

```http
POST /score/run-remote-task
```

评分完成后，可以通过以下接口读取结果：

```http
GET /score/remote-tasks/:taskId/result
```

当前已部署的评分服务地址为：

```text
http://8.136.155.63:3000
```

新增管理页面用于粘贴一份远端任务 JSON，对同一个用例连续发起 10 次评分。页面对每次评分使用递增后的 `taskId`，拉取 AI 评分结果后进行横向对比，并基于“多数结果稳定性”计算评分一致性。

## 目标

- 新增一个用于创建评分一致性分析任务的管理台页面。
- 支持在同一个页面创建多个一致性任务。
- 每个一致性任务通过输入框接收一段 JSON 字符串。
- 执行前解析并校验 JSON。
- 不要求用户提供 callback，提交时统一设置 `callback: ""`。
- 每个一致性任务固定创建 10 次远端评分运行。
- 每次运行使用递增后的 `taskId`。
- 提供手动刷新状态按钮，点击后批量读取 10 条远端任务的 registry 状态。
- 展示每次评分的对比结果，包括总分、规则不满足度和风险项。
- 使用多数结果稳定性计算一致性百分比。
- 第一版尽量保持轻量，主要在管理台前端完成；只有确实需要规避跨域或代理问题时再补后端代理。

## 非目标

- 不引入人工标注的标准答案。
- 不把一致性定义为人工评分准确率。
- 不修改评分引擎、评分提示词或结果生成逻辑。
- 不要求用户输入 callback URL。
- 一致性任务集合持久化到后端本地文件表，路径与 `remote-task-index.json` 同级。
- 不改变现有管理台分析页面语义。

### 任务创建

首页展示创建按钮，点击按钮拉起创建任务的抽屉，包含：

- 多行 JSON 输入框。
- 评分服务地址输入框，默认值为 `http://8.136.155.63:3000`。
- 固定运行次数展示，值为 `10`。
- `创建一致性任务` 按钮。
- JSON 格式错误或必填字段缺失时的校验错误区域。

JSON 输入期望符合远端任务结构：

```ts
type RemoteEvaluationTask = {
  taskId: number;
  testCase: {
    id: number;
    name: string;
    type: string;
    description: string;
    input: string;
    expectedOutput: string;
    fileUrl: string;
  };
  executionResult: {
    isBuildSuccess: boolean;
    outputCodeUrl: string;
    diffFileUrl?: string;
  };
  callback?: string;
};
```

页面只校验执行所需的必填字段。远端产物 URL 只校验是否为非空字符串，不在前端尝试访问或深度校验。

### 任务列表

每个创建出来的一致性任务在列表中展示：

- 页面内部的一致性任务 ID。

- 用例 ID 和用例名称。
- 创建时间。
- 执行进度，例如 `7 / 10 completed`。
- 任务状态：`running`、`completed`、`partial_failed` 或 `failed`。
- 至少有一个结果完成后展示一致性百分比。
- 查看详情、重新运行等操作按钮。

多个一致性任务互不影响。创建新任务不会清空旧任务。

任务列表分页展示，默认每页展示10条

## 任务 ID 策略

每个一致性任务从输入 JSON 中的原始 `taskId` 派生 10 个实际提交的 `taskId`。

页面为每个一致性任务生成一个数字种子：

```ts
taskSeed = baseTaskId * 100000 + taskSequence * 100 + 1
```

然后生成 10 个实际提交的 `taskId`：

```ts
submittedTaskId = taskSeed + runIndex
```

其中 `runIndex` 为 `0..9`。

示例：

- 输入 `taskId`：`1306`
- 页面任务序号：`1`
- 实际提交 ID：`130600101` 到 `130600110`

这样既满足递增要求，也能减少同一浏览器会话内多个一致性任务之间的 ID 冲突。如果生成结果不是安全整数，页面阻止创建任务，并提示用户使用更小的原始 `taskId`。

原始 `taskId` 保留在一致性任务摘要中用于追溯。实际提交时只替换顶层 `taskId`，并把 `callback` 设置为空字符串。

## 远端执行流程

每个一致性任务按以下流程执行：

1. 解析 JSON 字符串。
2. 校验必填字段。
3. 创建 10 条运行记录，并写入生成后的 `taskId`。
4. 提交每次运行：

```http
POST http://8.136.155.63:3000/score/run-remote-task
content-type: application/json
```

5. 点击“刷新状态”时批量读取 10 个派生任务状态：

```http
GET http://8.136.155.63:3000/score/remote-tasks/status?taskIds=130600101,130600102,...
```

6. 按响应解释运行状态：

- `preparing`、`queued`、`running`：页面按任务状态直接展示为进行中。
- `completed`：再读取单任务结果并解析总分、规则不满足度和风险项。
- `failed`、`timed_out`：标记该运行失败，并展示 registry 中记录的错误信息。
- `missing`：标记该运行未找到，提示远端任务登记缺失。
- 其他非 2xx：使用响应文本标记运行 `failed`。

页面应限制提交并发，避免压垮评分服务。第一版推荐行为：

- 10 次请求按顺序提交，服务侧已经实现并发设计。
- 页面不做自动轮询，不展示“轮询中”状态。
- 用户需要查看最新执行进展时，点击列表或详情页的“刷新状态”按钮。
- 支持失败任务自动重新提交。

## 结果提取

结果接口返回：

```ts
{
  success: true;
  taskId: number;
  status: string;
  resultData: unknown;
}
```

页面从 `resultData` 中提取以下字段：

- `overall_conclusion.total_score`
- `overall_conclusion.hard_gate_triggered`
- `overall_conclusion.summary`
- `rule_audit_results`
- `risks`
- `dimension_results`

不满足规则来自 `rule_audit_results` 中满足以下条件的项：

```ts
result === "不满足"
```

规则唯一标识使用：

```ts
rule_id
```

规则展示摘要优先级为：

```ts
rule_summary ?? conclusion ?? rule_id
```

风险项从 `risks` 中提取。用于跨运行对比的风险项标识应尽量稳定：

```ts
riskKey = normalize(level) + "|" + normalize(title)
```

如果 `title` 缺失，依次回退到 `description` 和 `id`。

规则不满足度计算方式：

```ts
unsatisfiedRuleCount / involvedRuleCount
```

其中 `involvedRuleCount` 统计结果不是 `不涉及` 的规则数量。如果不存在参与判定的规则，则规则不满足度为 `0`。

## 多数结果稳定性

一致性基于多数结果稳定性，而不是人工标准答案。

每个已完成运行先归一化成一个签名：

```ts
type RunSignature = {
  scoreBand: string;
  hardGateTriggered: boolean;
  unsatisfiedRuleKeys: string[];
  riskKeys: string[];
};
```

### 分数区间

分数区间用于避免微小分数抖动导致整体结果被判为不一致。

推荐分数区间计算：

```ts
scoreBand = Math.round(totalScore / 3) * 3
```

这表示大约 3 分以内的分数会被归入同一区间。

### 多数基准

多数基准只从已完成运行中计算：

- 多数分数区间：出现次数最多的 `scoreBand`。
- 多数硬门槛值：出现次数最多的布尔值。
- 多数不满足规则集合：至少出现在一半已完成运行中的规则。
- 多数风险项集合：至少出现在一半已完成运行中的风险项。

如果没有任何运行完成，则一致性不可用。

### 单次运行是否一致

一个已完成运行同时满足以下条件时，视为与多数基准一致：

- `scoreBand` 等于多数分数区间。
- `hardGateTriggered` 等于多数硬门槛值。
- 不满足规则集合与多数规则集合的相似度至少为 `0.8`。
- 风险项集合与多数风险项集合的相似度至少为 `0.8`。

集合相似度使用 Jaccard 相似度：

```ts
intersectionSize / unionSize
```

两个空集合的相似度定义为 `1`。

### 最终百分比

```ts
consistencyPercentage = consistentCompletedRuns / completedRuns * 100
```

失败或超时的运行不进入分母，但总览中单独展示失败数量。如果已完成运行少于 3 次，结论中需要提示样本数不足。

## 报表

### 规则不满足报表

对已完成运行聚合：

- 规则 ID。
- 规则摘要。
- 不满足次数。
- 不满足率。
- 第一条结论样例。
- 出现在哪些运行序号中。
- 稳定性状态。

稳定性状态：

- `稳定不满足`：不满足率 >= 80%。
- `稳定满足或不涉及`：不满足率 <= 20%。
- `判定波动`：其他情况。

### 风险项报表

对已完成运行聚合：

- 风险项 key。
- 风险等级。
- 风险标题。
- 出现次数。
- 出现率。
- 出现在哪些运行序号中。
- 证据样例。
- 稳定性状态。

稳定性状态：

- `稳定出现`：出现率 >= 80%。
- `偶发出现`：出现率 > 0 且 < 80%。
- `未出现`：出现率 = 0。

### 结论文案

页面生成简洁结论：

```text
本次 AI 评分结果一致性为 80%。10 次运行中 9 次完成，7 次与多数结果一致。主要波动来自 REQ-MUST-01 的规则判定和 2 个偶发风险项。
```

一致性等级：

- `>= 90%`：`评分结果高度一致`
- `70% - 89%`：`评分结果基本一致，存在少量波动`
- `< 70%`：`评分结果波动明显，建议人工抽查不稳定规则和风险项`



## 接口客户端设计

新增专用前端 helper，不把这部分逻辑混入现有 dashboard 列表 API：

```ts
submitRemoteScoreTask(baseUrl: string, payload: unknown): Promise<{ success: true; taskId: number; message: string }>;
fetchRemoteScoreResult(baseUrl: string, taskId: number): Promise<TaskResultResponse>;
```

helper 根据用户选择的评分服务地址拼接 URL，并去掉 `baseUrl` 末尾多余的 `/`。

本地开发时，`web/vite.config.ts` 可以补充 `/score/run-remote-task` 代理。但页面默认应能直接请求已部署的评分服务地址。

## 错误处理

- JSON 无法解析时禁用执行，并展示解析错误。
- 必填字段缺失时，在执行前列出缺失字段。
- `taskId` 不是数字或生成后不是安全整数时阻止创建任务。
- 某次提交失败只标记该运行失败，其他运行继续。
- 远端 registry 返回 `timed_out` 后标记该运行为超时。
- 结果 JSON 缺少预期字段时，运行仍可显示为完成，但缺失值展示为 `-`；如果无法读取总分，则该运行不参与一致性计算。
- 如果全部运行失败，页面不展示一致性百分比，并汇总主要错误信息。

## 测试策略

单元测试覆盖纯工具函数：

- JSON 校验。
- `taskId` 生成。
- 结果字段提取。
- Jaccard 相似度。
- 多数基准计算。
- 最终一致性百分比。
- 规则报表聚合。
- 风险项报表聚合。

前端组件测试覆盖：

- 创建多个一致性任务时不会清空已有任务。
- 正确渲染进度和总览指标。
- 失败运行和完成运行可以同时展示。

人工验证覆盖：

- 粘贴示例远端任务 JSON。
- 确认 10 个实际提交的 `taskId` 递增。
- 观察运行从已提交进入完成状态。
- 确认规则报表、风险项报表和单次运行详情一致。

## 已确定决策

- 一致性指多数结果稳定性，不是人工标签准确率。
- 页面支持在一个会话内创建多个一致性任务。
- 用户不需要提供 `callback`，提交时使用空字符串。
- `taskId` 由原始 `taskId` 派生，并生成递增的数字 ID。
- 一致性任务集合由后端文件表持久化，浏览器只负责读取和写回。
