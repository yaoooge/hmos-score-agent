# 模型供应商中立命名改造设计

## 1. 背景

当前工程已经接入了 Agent 辅助评分链路，但代码、配置、交互脚本、测试和文档中仍残留多处供应商指向命名，典型表现包括：

- 环境变量使用供应商指向命名，例如旧版的 `BASE_URL` / `API_KEY` / `MODEL` 品牌前缀形式
- 配置字段使用带品牌前缀的命名
- Agent client 类型使用品牌指向类名
- CLI 交互文案和 README 中直接提示品牌化的 `baseURL / apiKey`

这会带来两个问题：

1. 命名层面已经把实现绑定到单一供应商，不利于后续接入多家模型服务。
2. 即使底层协议兼容，开发者仍会误以为当前实现只支持某一家供应商，影响后续扩展与维护。

因此，本轮需要对工程进行一次彻底的中立命名改造，移除所有与单一模型供应商品牌直接绑定的命名与文案。

## 2. 目标

本轮目标如下：

1. 全局移除所有供应商品牌指向命名。
2. 使用统一的“模型供应商 / model provider”中立术语重命名配置、类型、类名、交互文案和文档。
3. 不保留旧环境变量兼容逻辑，运行时只接受新的中立环境变量。
4. 保留当前 Agent 辅助评分能力和现有真实验证通过的兼容接口行为。
5. 保证重命名后端到端评分链路仍可运行。

## 3. 非目标

本轮明确不做以下事项：

- 不新增第二家模型供应商接入实现
- 不改造为多 provider 动态路由框架
- 不改变当前 Agent 辅助评分的工作流结构
- 不改变已有评分结果 schema 中与供应商无关的字段
- 不保留旧命名的向后兼容层

## 4. 设计原则

### 4.1 命名中立优先

凡是“供应商名称只是历史偶然，不应成为业务概念”的位置，全部替换为中立命名。

### 4.2 只保留协议语义，不保留品牌语义

当前 client 本质上是“遵循 chat completions 协议的模型服务客户端”，因此应直接使用协议导向的中立命名，而不是带历史迁移语义的命名。

### 4.3 一次切干净

用户已明确要求“彻底切掉”，因此：

- 不读取旧环境变量
- 不保留旧类名别名
- 不在 README 中继续出现旧示例

### 4.4 运行能力不退化

虽然是命名改造，但必须保证：

- 本地测试通过
- 构建通过
- 真实评分 CLI 继续可运行
- 当前 chat completions 接口的请求与输出归一化能力不回退

## 5. 总体方案

本轮改造分四层进行。

### 5.1 配置层重命名

将环境变量统一替换为：

- `MODEL_PROVIDER_BASE_URL`
- `MODEL_PROVIDER_API_KEY`
- `MODEL_PROVIDER_MODEL`

对应配置字段统一替换为：

- `modelProviderBaseUrl`
- `modelProviderApiKey`
- `modelProviderModel`

所有读取 `.env`、写入 `.env`、运行时读取 `process.env` 的逻辑同步调整。

### 5.2 Agent Client 层重命名

将当前供应商指向类名：

- 旧版品牌指向的 agent client 类名
- 旧版品牌指向的 options 类名

替换为中立命名：

- `ChatModelClient`
- `ChatModelClientOptions`

`AgentClient` 接口名称保持不变，因为它本身已经是业务抽象，不带供应商耦合。

### 5.3 交互与落盘文案重命名

将所有用户可见文案中的品牌字样替换为中立表达，例如：

- `模型服务 baseURL`
- `模型服务 apiKey`
- `模型服务 model`
- `模型供应商配置`

`case-info.json` 中已有的 `agent_model` 字段保留，因为它描述的是“Agent 使用的模型标识”，不是供应商品牌。

### 5.4 测试与文档重命名

同步更新：

- 单元测试
- workflow 测试
- README
- 当前仍可能指导后续实现的设计文档和计划文档

确保仓库中不再残留供应商品牌指向命名。

## 6. 目标命名规范

### 6.1 环境变量

统一使用：

- `MODEL_PROVIDER_BASE_URL`
- `MODEL_PROVIDER_API_KEY`
- `MODEL_PROVIDER_MODEL`

### 6.2 配置字段

统一使用：

- `modelProviderBaseUrl`
- `modelProviderApiKey`
- `modelProviderModel`

### 6.3 类型与类名

统一使用：

- `ChatModelClient`
- `ChatModelClientOptions`

### 6.4 交互层字段

交互式脚本内部字段可保持简洁：

- `baseUrl`
- `apiKey`
- `model`

避免保留任何品牌前缀。

## 7. 受影响范围

### 7.1 代码

预计至少会修改以下代码区域：

- `src/config.ts`
- `src/service.ts`
- `src/agent/agentClient.ts`
- `src/tools/runInteractiveScore.ts`
- `src/workflow/scoreWorkflow.ts`

### 7.2 测试

预计至少会修改：

- `tests/interactive-launcher.test.ts`
- `tests/agent-client.test.ts`
- 其他引用旧环境变量或旧类名的测试

### 7.3 文档

预计至少会修改：

- `README.md`
- 当前仍会被开发参考的 design / plan 文档

## 8. 方案比较

### 方案 A：彻底中立化

做法：

- 统一改为 provider / model provider 命名
- 不保留旧兼容

优点：

- 最符合长期演进方向
- 没有新旧术语并存
- 后续接入多家模型时认知成本最低

缺点：

- 需要同步修改配置、测试、文档，改动面较大

### 方案 B：保留 “compatible” 但中立品牌

做法：

- 仅把品牌词去掉，保留协议兼容语义

优点：

- 既中立又保留当前实现的真实技术语义

缺点：

- 若未来不再局限 chat completions 接口，还会再做一轮抽象

### 方案 C：只改用户可见文案

做法：

- 内部代码暂时不动

优点：

- 改动最小

缺点：

- 无法真正解除供应商耦合
- 会继续误导后续开发

本轮采用：

- 方案 A 为主
- 方案 B 的“compatible”语义仅保留在 client 类型名中

## 9. 测试策略

### 9.1 单元测试

覆盖：

- 配置层只读取新的 `MODEL_PROVIDER_*`
- client 类型与构造逻辑完成重命名
- 交互式写 `.env` 只写入新的 provider 环境变量

### 9.2 回归测试

覆盖：

- 现有 agent 辅助评分链路不因重命名损坏
- 兼容接口降级重试逻辑仍然有效
- provider 兼容 JSON 归一化仍然有效

### 9.3 端到端验证

至少执行：

- `npm test`
- `npm run build`
- `npm run score -- --case init-input`

并确认落盘结果、日志和输入快照中不再出现旧命名。

## 10. 风险与控制

### 10.1 `.env` 断档风险

由于本轮不保留旧变量兼容，已有本地环境如果仍是旧版品牌变量，会直接失效。

控制方式：

- README 和交互脚本全部同步切换
- 测试覆盖 `.env` 更新逻辑

### 10.2 文档残留风险

若代码已改完但文档还保留旧名，会继续误导后续开发。

控制方式：

- 把 README、设计文档、计划文档一起纳入改造范围
- 使用全局搜索确认无残留

### 10.3 机械改名误伤风险

若只做字符串替换，容易漏掉类型引用或测试断言。

控制方式：

- TDD 先补失败测试
- 最后以 `rg` 全局扫描残留字符串
- 再执行测试和真实 CLI 验证

## 11. 实现范围建议

当前一轮实现范围控制为：

1. 统一切换新的 provider 环境变量
2. 统一切换配置字段与 client 类型命名
3. 更新交互脚本、README、测试
4. 更新当前仍具指导意义的设计/计划文档
5. 跑完测试、构建和真实评分 CLI

这样可以在不扩展新 provider 能力的前提下，把工程的供应商耦合命名一次切干净。
